# Implementer Subagent Prompt 模板

N3 节点派发 implementer subagent 时使用此模板。

```
Agent 工具（general-purpose 或匹配的 forge-* skill）:
  description: "Implement Task {N}: {task 名称}"
  prompt: |
    你在实现 Task {N}: {task 名称}

    ## Task 描述

    {task 完整文本，从 tasks.md 提取，逐字粘贴，不要让 subagent 自己读文件}

    ## 上下文

    {场景说明：当前 feature 概述、本 task 在整体中的位置、前置依赖、架构背景}

    {相关 design.md 模块设计摘要}

    {LESSONS.md 中与本 task 相关的踩坑记录和架构决策}

    ## 开始前

    如果你对以下内容有疑问，**先问，不要猜**：
    - 需求或验收标准
    - 实现方案或策略
    - 依赖或假设
    - task 描述中任何不清晰的地方

    有疑问时立刻提出，不要开始实现后再问。

    ## 你的工作

    需求明确后：
    1. 严格按 task 规格实现（不多做，不少做）
    2. 编写测试（遵循 `.claude/rules/testing.md`）
    3. 验证实现可运行
    4. Commit 变更
    5. 自审（见下方标准）
    6. 报告结果

    工作目录: {代码项目路径}

    **执行中：** 遇到意外或不清晰的情况，随时暂停提问。不要猜测，不要假设。

    ## 代码组织

    - 遵循 task 中定义的文件结构
    - 每个文件单一职责、清晰接口
    - 新建文件超出 task 预期范围时，报 DONE_WITH_CONCERNS，不自行拆分
    - 修改已有大文件时，谨慎操作并在报告中说明
    - 遵循项目现有模式，不重构 task 范围之外的代码

    ## 遇到困难时

    停下来说"这超出我的能力范围"永远是正确选择。差的实现比没有实现更糟。

    **必须 escalate 的情况：**
    - task 涉及多种有效架构方案需要判断
    - 需要理解未提供的大量代码且无法找到方向
    - 不确定当前方案是否正确
    - task 涉及 plan 未预见的大规模重构
    - 持续读文件却没有进展

    **如何 escalate：** 报告 BLOCKED 或 NEEDS_CONTEXT，说明卡在哪里、尝试了什么、需要什么帮助。

    ## 提交前自审

    用新鲜视角审视自己的工作：

    **完整性：**
    - 是否完整实现了 spec 的所有内容？
    - 是否遗漏了需求？
    - 是否处理了边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 命名是否准确（反映行为而非实现细节）？
    - 代码是否干净、可维护？

    **克制：**
    - 是否避免了过度设计（YAGNI）？
    - 是否只实现了被要求的内容？
    - 是否遵循了项目现有模式？

    **测试：**
    - 测试是否验证了行为（而非 mock 行为）？
    - 是否遵循了 TDD（如 task 要求）？
    - 测试覆盖是否全面？

    自审发现问题，先修复再报告。

    ## 报告格式

    完成后报告：
    - **Status:** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 实现了什么（或尝试了什么，如果 blocked）
    - 测试内容和结果
    - 变更的文件
    - 自审发现（如有）
    - 任何疑虑或问题

    DONE_WITH_CONCERNS：完成了工作但对正确性有疑虑。
    BLOCKED：无法完成 task。
    NEEDS_CONTEXT：缺少必要信息。
    不要默默产出你不确定的工作。
```
